原以为自己对于js的词法作用域理解的还不错，直到遇到一大波类似于上述的奇奇怪怪的现象后，我就有些凌乱了，成果如下：

作用域

变量作用域

变量作用域scope可以理解为能够访问到这个变量的所有区域，例如可以在全局范围内访问到全局变量，那么全局变量的scope就是全局作用域。

函数作用域

对于函数来讲，任何在函数内部定义的变量、函数的参数、在函数内定义的其他函数都是局部变量。只有在函数的内部才能访问到这些局部变量。
函数作用域：函数的内部变量其scope是这个函数体以及这个函数体内嵌套的任意函数体内部。
声明提前：由函数作用域知内部变量在函数体内任何位置都是可访问的，这就导致有可能变量在声明前就是可访问的。在js中函数内声明的所有变量其声明过程都被提前到函数体的顶部，而赋值过程仍在代码中原位置。这时由JS引擎的“预编译”导致的。
内部变量的优先级：同名的内部变量的优先级要高于全局变量，这也可由作用域链可知。

js的词法作用域链

当函数在定义时，就保存了一个作用域链。作用域链确保了可访问的变量和函数，以及访问顺序。作用域链式在代码定义时就已确定的，而不是等到执行时才确定。

变量解析

沿着作用域链前端的第一个对象开始查找，一旦找到就返回。

预编译

对预编译的理解

js在执行前会对代码进行预编译，关于预编译需要注意几点：

  • 预编译工作范围：对var变量声明、以及function(){}函数体内的代码进行处理，并不进行赋值操作。赋值操作发生在执行阶段。举例说明三种情况：
    • var val=1;预编译只做了声明的工作即var val;
    • function f(){/函数体代码段}遇到这样的函数定义，会对函数体内代码段进行预编译
    • var f=function(){//函数体代码段}这种情况严格意义上讲不属于函数定义，称为对变量f进行声明和初始化比较合适，因此预编译只进行变量f的声明即var f;，对匿名函数体内代码段不进行预编译。

  • js的代码是顺序分块进行预编译-执行 的过程的。顺序是指按照代码定义顺序从上到下进行，分块是针对有多个<script type="text/javascript"></script>块时，按顺序将第一个<script></script>块内代码预编译然后执行，再对下一个<script></script>重复该预编译-执行的过程。

  • 由上条可知，当有多个代码块时，后面出现的<script></script>代码块能够访问前面出现的<script></script>代码块内的全局变量，反过来则不行。

  • 对函数内部代码进行预编译时，会忽略逻辑判断。例如if(false){var val=1;}仍然会进行var val;预编译。

继承（原型链）

在js中继承主要是靠原型链实现的。
在js中，创建一个函数（包括构造函数），就为该函数赋一个prototype原型属性，该属性指向原型对象。该原型属性默认获得 一个constructor属性，指回构造函数，原型属性还继承了Object的属性和方法（toString，valueOf等）。通过该构造函数创建的所有实例都有一个[[prototype]]属性，指向原型对象。这样原型中所有的属性和方法就为所有实例共享。

原型链继承

从以上所说的构造函数、原型对象、实例之间的关系就很好理解原型链继承。原型链继承实质就是使一个对象B的原型对象为另一个对象A的实例。这样对象B的实例指向B的原型对象，B的原型对象又指向A的原型对象，形成了原型链，而B的实例也继承了原型链上的所有属性和方法。

对象中属性的读取

当读取对象的一个属性时，实际是进行了一次搜索。搜索先从实例本身开始，若找到则停止搜索返回其值；若未找到，则继续搜索其[[prototype]]指向的原型对象，若找到返回，未找到返回undefined。

js中数值和字符串的一点事儿

发表于 2013-10-03

写在前面

有让你觉得意外的地方么~

数值、字符串大小比较

  • 字符串之间：这个比较简单，逐个字符进行比较ASCII码值。

  • 字符串与数值之间：将字符串转换数值，然后比较数值大小。 

    alert(10 < "11c");//false
alert(10 > "11c");//false

    上述代码两次均返回false，这是由于"11c"不能转化为数值。可以测试isNaN("11c")返回true。
    数值与NaN比较大小永远返回false。
    需要注意的是，此处字符串转换为数值的规则与parseInt()是不相同的。此处的规则与Number()和isNaN()是一致的，规则如下:

    • 若字符串只包含数字、正号+ 和负号-，转化为十进制。例如： 

      alert(+"10" > 9);//true 即(10<9) 
 alert(+"10>"9");//true 即(10>9)
   alert(-"10"<9);//true 即(-10<9)
  alert("-10"<9);//true 即(-10<9)
    • 若包含有效的浮点格式，例如2.1，则转化为响应的浮点数
    • 若为空字符串，则转换为0

    • 若包含0x则转换为十六进制，若包含0则不按照八进制解析而忽略头部的任意多个0； 

      alert("011"<9)//false 即(11<9)  
alert("0x11"<15)//false 即(17<15)

    • 若包含以上格式其他字符，则返回NaN，例如 

      alert(10 < "11c");//false 即(10<NaN)
   alert(10 > "11c");//false 即(10>NaN)

parseInt()工作原理

parseInt(string,radix)用法：

  • 返回值是十进制；
  • 无视string中最前和最后的空格；
  • 参数radix是指解析string的进制，若未指定radix或值为0则按默认进制对string进行解析；

  • 未指定radix的情况下，若string的首字符为0，则按照八进制进行解析；若string的首字符为0x则默认按照十六进制解析string，其他情况默认为十进制； 

    parseInt("011");//9
parseInt("0x11");//17

  • 解析从第一个字符开始，到第一个不能解析的字符（空格、英文、标点等）结束；重点说一下，能解析的字符包括数字、正号+和负号-； 

    parseInt("1a");//1
parseInt("");//NaN
parseInt("-1");//-1

  • 若第一个字符就不能解析，则返回NaN；

实例一： 

var c = parseInt("017/08/2009");
alert(c);               //15

解析：未指定radix，第一个字符为0，则按照八进制进行解析，且解析到7停止，017输出15。

实例二： 

var d = parseInt("09/08/2009");
alert(d);              //0

解析：未指定radix，第一个字符为0，按照八进制解析时9属于不能解析的字符，故0输出0.

Number()和parseInt()区别（实例）

Number("1a");//null
parseInt("1a");//1

Number("");//0
parseInt("");//NaN

Number("011");//11
parseInt("011");//9

CSS盒模型（box model）

发表于 2013-09-16

CSS盒模型（box model）

CSS盒子模型就是指对于页面上的每一个元素（无论是块级元素还是行内元素）都被看做一个矩形框，由元素内容、内边距、边框、外边距组成。盒子决定了该元素在页面中的占位。值得一提的是标准盒子模型与混杂盒子模型是有差别的。
标准盒子模型的实际占位：width+padding*2+border*2+margin*2
混杂盒子模型的实际占位：width+margin*2

如何解决混杂盒子模型带来的不便：
我们可以避免混杂模式的触发。通过文档类型声明DOCTYPE，可以选择是触发混杂模式还是标准模式。通过设置文档类型声明，可以告诉浏览器使用哪种文档类型定义DTD，以及选择哪种HTML和网页呈现的方式。当页面以混杂模式呈现时，其行为是不可预知的，所以要尽量避免混杂模式的触发。混杂模式通常是为了模拟老版的浏览器使其仍能够正常运行。

注：在CSS2中背景应用于内容和内边距组成部分；而在CSS2.1中背景已经延伸到了边框（这也是CSS标准规定的）

在CSS3中这个是可设的。

内边距

内边距的值可以是百分比，是相对与其父元素的宽度width来计算的，不管是垂直内边距或水平内边距都是根据父元素的width计算的。

边框

需要注意的是，如果你希望有边框就一定要记得设置样式border-style，因为默认是none，即便设置了边框宽度和颜色也不会显示的。

若没有设置边框颜色，那么边框颜色默认为元素文本的颜色，若当前元素无文本那么边框颜色就是其父元素的文本颜色。

外边距

和内边距一样，给外边距设置值为百分比时，是相对于其父元素的宽度width来计算的。
对于行内元素来讲，垂直的外边距不会产生效果。

  • dragstart、drag、dragend这三个事件是针对被拖拽元素的
  • dragenter、dragover、dragleave、drop这三个事件时针对目标元素的

阻止默认行为很关键

  • 在目标行为的drop事件处理程序中，阻止默认行为，（因为默认不接受被拖放的元素）
  • 对整个页面的dragover、drop事件，也要阻止默认行为，否则拖放处理不能够实现。

CSS布局之定位

发表于 2013-09-06

CSS中定位方案

有三种：标准流、浮动、绝对定位

display

块框（block box）、行内框（inline box）、行框（line box）
HTML中一切元素都会形成一个”框”(box)，块级元素即块框，行内元素即行内框。由一行形成的水平框即行框，行框可能包含多个行内框。行框的高度总是足以容纳它包含的所有的行内框，他们之间的关系如下图所示：

块级元素、行内元素的区别

常见的块级元素：<h1>~<h6>、<p>、<ol>、<ul>、<dl>、<dt>、<li>、<table>、<form>、<caption>、<div>
常见的行内元素：<span>、<img>、<input>、<em>、<srong>等

  • 块级元素：
    独占一行，默认情况下，宽度会占满父元素，多个块级框从上到下排列；垂直距离根据垂直外边距计算；
    可以设置width、height
  • 行内元素：
    不独占一行，多个块级框在一行中水平布置，可以通过内边距、外边距调整行内框之间距离；
    设置width、height无效；
    其padding、margin属性只在水平方向 有效，在垂直方向无效。

position

  • static：
    元素仍然存在与标准流中。元素框正常生成。
  • relative：
    元素框在标准流中的占位保留；
    元素的display属性不变；
    元素的top、bottom、left、right、z-index属性被激活；
    元素的top、bottom、left、right都是相对于自身在标准量中的占位而言；
  • absolute：
    元素框脱离标准文本流，不再占位；
    元素的display属性被改变为inline-block，而不管之前该元素是块级元素还是行内元素
    元素的top、bottom、left、right、z-index属性被激活；
    元素的top、bottom、left、right都是相对于其设置了postion:relative或position:absolute的最近祖先元素而言；
  • fixed：
    其表现与absolute相似，可以说是absolute的一种，区别就在于fixed的top、bottom、left、right定位是相对于视窗而言。
    元素框脱离标准文本流，不再占位；
    元素的display属性被改变为inline-block，而不管之前该元素是块级元素还是行内元素
    元素的top、bottom、left、right、z-index属性被激活；

特别注意： top的优先级高于bottom，left的优先级高于right；当设置矛盾时会优先top和left的设置，这个在relative、absolute、fixed下都是适用的，如下：

z-index优先级：

  • 不显示设置z-index时，relative、absolute、fixed优先级一样，后定义的元素出现在上层
  • 显示设置z-index时，该属性值大的在上层。

    display与position的关系

    上面说过，position:absolute\fixed会将元素的display改变为inline-block，这时候如果显示设置display为其他值会失效。

同样，float也会将元素的display改变为inline-block，这时候如果显示设置display为其他值也会失效。

BFC和hasLayout和containing block

大多时候一个元素的尺寸和位置不是由自己决定的（例如：未设置宽、高尺寸的块级元素、绝对定位的元素等），而是要参考它的某个祖先容器。那详细的问题就来了：什么时候一个元素能够当家做主决定自己的尺寸和位置呢？当自己不能决定时要参考那个祖先元素呢？

首先引入IE的布局概念，hasLayout（拥有布局）的元素能够负责自身的以及子元素的尺寸和位置设置。当一个元素没有布局不能决定自身尺寸和位置时，由离它最近的hasLayout（拥有布局）的元素来决定。举例说明吧： 

<style type="text/css">
      .father
      {
          border:1px solid #ccc;
          float:left;
      }

</style>    
<body>
    <div class="test">
            局概念
    </div>
</body>

我们看到上面代码中的元素.test没有设置宽、高，它的尺寸由父body（在IE中body是默认hasLayout的元素之一）来决定，所以可以看到.father的宽度是与父等宽的，而不管其中元素是否撑满。

然后给.father设置float:left，你会发现此时.father的宽度根据其内容进行调整，而不再是与其父元素等宽。这是由于float：left使.father拥有布局(hasLayout)。

在上面那个例子已经谈到了hasLayout和containing block：

haslayout：一个元素如果hasLayout那么能够决定自己的尺寸和位置，对于没有布局的元素，可以通过一些CSS设置使其hasLayout：

  • 设置width
  • 设置height
  • 使其浮动float
  • 设置display为inline-block
  • 设置overflow为hidden、auto、scroll
  • 设置zoom(仅IE)

containing block(包含块)

上面说到，当一个元素E无法决定自己的尺寸和定位，需要依靠其某一祖先元素来决定，这里提到的某一个祖先元素就是E的containing block(包含块)：

  • 初始包含块：即浏览器窗口
  • 对于设置position:fixed的元素E，其位置需要依靠浏览器窗口决定个，因此E的包含块就是初始包含块
  • 对于设置position:absolute的元素E，其包含块就是’离E最近的设置了position:relative\absolute的祖先元素’

BFC

我觉得BFC在主流浏览器中的角色类似于hasLayout在IE中的功能。
